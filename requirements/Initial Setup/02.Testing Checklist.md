## 1. Environment Setup

> Do this once before running any milestone tests.

### 1.1 Prerequisites
- [x] Docker Desktop is running
- [ ] Python 3.13 is installed (`py -3.13 --version`)
- [ ] Node.js 20+ is installed (`node --version`)
- [ ] Expo CLI is available for the mobile app (`npx expo --version`)
- [ ] A terminal is open at `GigKraft/dev/`

### 1.2 Start Infrastructure
```bash
# From GigKraft/dev/
docker compose up -d postgres redis
docker compose ps
```
- [x] `postgres` container shows status `healthy`
- [x] `redis` container shows status `healthy`

### 1.3 Backend Setup
```bash
# From GigKraft/dev/backend/
py -3.13 -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
python manage.py migrate
python manage.py seed_data
python manage.py runserver
```
- [ ] `pip install` completes without errors
- [x] `migrate` runs without errors
- [x] `seed_data` creates sample node, pros, homeowners, krafts
- [ ] Server starts and API docs load at `http://127.0.0.1:8000/api/docs`
- [ ] API health endpoint returns `ok` at `http://127.0.0.1:8000/api/health`
- [ ] Do not use `http://127.0.0.1:8000/` as a test target; the backend root path is not configured

### 1.4 Frontend Setup
```bash
# From GigKraft/dev/frontend/
npm install
copy .env.example .env
npm run generate:api
npm run dev
```
- [ ] `npm install` completes without errors
- [ ] `generate:api` produces `src/api/generated/types.ts` from the backend OpenAPI spec
- [ ] Dev server starts at `http://localhost:5173`
- [ ] Frontend is started with `npm run dev` (Vite), not `npx expo start`

### 1.5 Mobile Setup
```bash
# From GigKraft/dev/mobile/
npm install
copy .env.example .env
npm run generate:api
npx expo start
```
- [ ] `npm install` completes without errors
- [ ] `generate:api` produces `src/api/generated/types.ts`
- [ ] For Android emulator testing, mobile `.env` uses `EXPO_PUBLIC_API_BASE_URL=http://10.0.2.2:8000`
- [ ] For physical device testing, mobile `.env` uses the computer's LAN IP instead of `localhost`
- [ ] Expo starts, press `w` to open web preview or scan QR for device

## 3. Milestone 2 — Backend API

### 3.1 Health and Docs
- Open `http://127.0.0.1:8000/api/health`
- [ ] Returns `{ "status": "ok" }` (or similar)
- Open `http://127.0.0.1:8000/api/docs`
- [ ] Interactive Ninja docs page loads

### 3.2 User Registration
```
POST http://127.0.0.1:8000/api/auth/register
Body: { "email": "pro1@test.dev", "password": "Test1234!", "role": "pro" }
```
- [ ] Returns 201 with `access_token` and `refresh_token`
- [ ] Role in token payload is `pro`

```
POST http://127.0.0.1:8000/api/auth/register
Body: { "email": "home1@test.dev", "password": "Test1234!", "role": "homeowner" }
```
- [ ] Returns 201 with tokens, role is `homeowner`

### 3.3 Login
```
POST http://127.0.0.1:8000/api/auth/login
Body: { "email": "pro1@test.dev", "password": "Test1234!" }
```
- [ ] Returns `access_token` and `refresh_token`

### 3.4 Token Refresh
```
POST http://127.0.0.1:8000/api/auth/refresh
Body: { "refresh_token": "<token from login>" }
```
- [ ] Returns new `access_token`
- [ ] Using a wrong/expired token returns 401

### 3.5 Protected Endpoint
```
GET http://127.0.0.1:8000/api/me
Header: Authorization: Bearer <access_token>
```
- [ ] Returns user profile with correct role
- [ ] Without token returns 401

### 3.6 Mock OTP Auth
```
POST http://127.0.0.1:8000/api/auth/otp/request
Body: { "phone": "+12025550001" }
```
- [ ] Returns `{ "dev_code": "123456" }` in mock mode

```
POST http://127.0.0.1:8000/api/auth/otp/verify
Body: { "phone": "+12025550001", "code": "123456" }
```
- [ ] Returns tokens and creates user with `pro` or `homeowner` role

### 3.7 Mock Google Auth
```
POST http://127.0.0.1:8000/api/auth/google
Body: { "token": "mock-google:manager@test.dev" }
```
- [ ] Returns tokens and creates/returns `node_manager` role user

- [ ] All tests pass

## 6. Milestone 5 — Core MVP Domain Flows

### 6.1 Kraft Guardrail (Backend)

**Test: Publish blocked without After photo**
```
POST /api/krafts  (create a kraft with no photos)
POST /api/krafts/{id}/publish
```
- [ ] Returns error: "A Kraft requires at least one After photo"

**Test: Publish blocked without confirmed invoice**
```
POST /api/krafts  (create kraft with after photo, invoice_confirmed=false)
POST /api/krafts/{id}/publish
```
- [ ] Returns error: "A Kraft requires a confirmed invoice cost"

**Test: Publish succeeds with both**
```
POST /api/krafts  (after photo + invoice_confirmed=true + invoice_cost set)
POST /api/krafts/{id}/publish
```

- [ ] Admin can verify it → status becomes `verified`
